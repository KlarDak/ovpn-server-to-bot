Вторая версия коннектора от сервера с `OpenVPN` к миром посредством стандарных `HTTP`-методов, реализованных по принципам `REST API`.

## Настройка

Для работы системы необходимы следующие дополнения:

1. ``Redis`` - для работы системы хранения временных ссылок.
2. ``sqlite3`` - для хранения данных пользовательских конфиг-файлов.
3. ``OpenVPN`` - сам OpenVPN.
4. ``easyrsa`` - генератор ключей для работы OpenVPN.

### Установка библиотеки

Используйте следующую команду:

```bash
git clone https://github.com/KlarDak/ovpn-server-to-bot
```

После клонирования репозитория, перейдите в него и установите зависимости:

```bash
cd ovpn-server-to-bot/

npm install
```

На этом установка всего необходимого для работы системы будет завершена.

## Запуск

> Перед запуском убедитесь, что файлы ``.env`` и ``.env.server`` созданы и правильно заполнены. Без этого система **НЕ запустится**.

Запуск ``dev``-версии для работы на базе ``.ts``-кода:

```bash
npm run dev
```

Запуск скомпилированной версии (``.js``-файлов):

```bash
npm run
```

Запуск тестов системы:

```bash
npm run test
```

## Скрипты обработчика OpenVPN

Директория ``scripts`` содержит скрипты для работы ``OpenVPN``: работа с конфиг-файлами, подключение/отключение пользователей + инсталлер и система резервного копирования ключей.

Для старта работы конфиг файлов сделайте файлы исполняемыми.

```bash
sudo chmod +x *.sh
```

Для работы ``inconn.sh`` необходимо подключить **в конфиг-файле сервера**. Внесите в него следующее значение:

```text
...
script-security 2
client-connect [сюда абсолютное местоположение файла]
...
```

Скрипт ``auto_installer.sh`` используется однократно, для установки сервера. Если сервер уже установлен, его можно использовать для настройки конфиг-файла. 

Для настройки, используйте следующее:

```bash
sudo ./auto_installer.sh config IPADDR PORT PROTOCOL DIR
```

Где:
1. ``config`` - команда на генерацию конфигурации сервера.
2. ``IPADDR`` - хостовой IP-адрес системы.
3. ``PORT`` - порт OpenVPN.
4. ``PROTOCOL`` - протокол работы OpenVPN (``upd`` или ``tcp``).
5. ``DIR`` - директория хранения файлов ``OpenVPN``.

## Детали о системе безопасности

В системе работы с сервером используются некоторые решения на базе ``.env``-файла и полезная нагрузка JWT-токена, призванные усилить систему безопасности сервера.

### Секретный ключ

Значение ``SECRET_KEY`` - секретный ключ для шифрование JWT-токена. Обязательно сделайте его максимально сложным для предовращения компрометации ключей.

### Доступные IP-адреса

Значение ``ALLOWED_IPS`` - доступные для работы IP-адреса. **Обязательно для заполнения**. Используется для ограничение выборки доступных IP-адресов для работы с системой. Соединения с адресов, отсутствующих в списке, будут обрываться.

### Типы в JWT-токене

Для разделения обработчиков используются **роли и типы**. Вся система автоматически настраивает необходимые данные.

**Пример токена:**
```json
{
  "sub": "x",
  "aud": "y",
  "iat": 2769553370,
  "exp": 2769653370,
  "role": "a",
  "type": "b"
}
```

Поля ``sub`` и ``aud`` представляют из себя буквенный индекс отправителя и получателя соответственно, то есть сервер-отправитель и сервер-получатель. Их нужно заполнить для подтверждения регистрации сервера в системе и раздделения данных между кластерами.

> **Важно:** данная информация указана для понимания системы. Все поля во время работы заполняются автоматически и с использованием данных файла `.env`. 

## Доступные запросы

Метод общения с сервером - ``REST API``, а значит для запросов настроены **стандартные HTTP-методы**.

### Работа с конфиг-файлами со стороны пользователя

Для работы пользователя с его конфиг-файлами, используется метод ``/v2.0/users/config``.

Передаются:
1. ``UUID`` - идентификатор конфиг-файла.
2. ``type`` - тип пользователя конфиг-файла.
3. ``time`` - время работы конфиг-файла (в секундах).

### Работа с конфиг-файлами 

Для работы админа с файлами пользователями, используется метод ``/v2.0/bot/config``.

> **Важно:** для управление конфиг-файлами используется **только** ``POST``-метод. Для разделения используется поле ``type`` в полезной нагрузке JWT-токена.

Передаются:
1. ``UUID`` - идентификатор конфиг-файла.
2. ``type`` - тип пользователя конфиг-файла.
3. ``time`` - время работы конфиг-файла (в секундах).
4. ``ban`` - блокировка конфиг-файла.

### Работа с подключёнными пользователями

Для работы с подключёнными пользователями используется ``/2.0/bot/active/``. Все запросы выполняются только через метод ``POST``. 


## Доступные ответы пользователей

Шаблонный ответ следующий:

```json
{
    "code": 200,
    "data": null,
    "message": "User pardoned successfully."
}
```

Где:
1. ``code`` - числовой HTTP-код ответа.
2. ``message`` - текстовое описание ответа.
3. ``data`` - полезная нагрузка ответа (необязательна).

Поле ``data`` может иметь разное содержимое. Специальная библиотека для работы с системой адаптирована под все виды ответов системы.

> Актуально на момент версии **dev-2.0.5.2** и **prod-2.0**. В будущем все данные об ответах и функций будут содержатся в файле **UTILITIES.md**.
