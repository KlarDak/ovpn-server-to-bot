Вторая версия коннектора от сервера с `OpenVPN` к миром посредством стандарных `HTTP`-методов, реализованных по принципам `REST API`.

## Настройка

Для установки проекта введите следующую команду:

```bash
npm install
```

`npm` автоматически произведёт установку всех необходимых дополнительных библиотек. 


## Настройка переменных окружения

Для работы проекта используются переменные окружения, устанавливаемые файлом `.env`.

Все необходимые переменные с описанием указаны в файле `.env.temp`. Перед запуском сервера необходимо заполнить и переименовать файл в `.env`.

**Важно:** убедитесь, что порты, указанные в полях ``PORT`` и ``REDIS_PORT``, открыты для входящих и исходящих соединений.

## Запуск сервера

Для запуска в состоянии ``prod`` запустите:

```bash
npm run
```

Будет запущен скомпилированный в Javascript код, который является на данный момент полностью стабильным.

В случае необходимости запуска нестабильной версии (в состоянии разработки), введите следующее:

```bash
npm run dev
```

## Методы обращения по API

Генерация конфиг-файлов работает в **автоматическом режиме**. Вся работа с созданием, редактированием, получением и удалением конфиг-файлов выполняется с помощью ``HTTP-методов``.

```http
/v2.0/users/config/
```

Следующие методы отвечают за следующие функции работы с конфиг-файлами:

``GET`` - получить статистику по конфиг-файлу.

**Важно:** в этом случае к запросу добавляется аргумент ``uuid``.

```http
/v2.0/users/config/[uuid]
```


``POST`` - сгенерировать новый конфиг-файл (автоматически сгенерируется пользовательский файл)

В тело запроса передаются **следующие параметры**:

1. ``uuid`` - UUID-идентификатор конфиг-файла пользователя.
2. ``type`` - тип конфигурационного файла.
3. ``time`` - время работы конфиг-файла (в секундах).

``PATCH`` - частичное обновление конфигурационного файла, согласно тому же UUID.

В тело передаются **те же** параметры, что и в предыдущем запросе, лишь за одним отличием:

1. ``uuid`` - UUID-идентификатор конфиг-файла пользователя **(ОБЯЗАТЕЛЬНЫЙ ПАРАМЕТР)**.
2. ``type`` или ``time`` - передаётся тип конфигурационно файла ИЛИ время работы конфиг-файла, или и то, и другое.

``PUT`` - полное обновление конфигурационного файла, согласно тому же UUID.

В тело передаются **все параметры**, указанные к методу ``POST``.

``DELETE`` - удаление конфигурационного файла и файла пользователя.

**Параметр:**
1. ``uuid`` - UUID-идентификатор конфиг-файла пользователя.

## Токен аутентификации

В качестве авторизационного метода используется токен JWT. Система уже настроена, необходимо лишь указать ключ для шифрования/дешифрования в файле ``.env`` (поле ``SECRET_KEY``).

**Важно:** указанный в поле ключ необходимо использовать и для дешифрования.

### Payload-данные токена

Отдельный проект для работы с библиотекой под базой Telegram-бота или веб-сайта на PHP уже содержит функции, предназначенные для автоматического формирования верного ``payload`` у JWT-токена, однако некоторые заголовки требуют уточнения.


**Пример строго типизированной payload-нагрузки:**
```json
{
  "sub": "ksd_sp_01",
  "aud": "ksd_nl_01",
  "iat": 1769553370,
  "exp": 1769653370,
  "role": "bot",
  "type": "check"
}
```

**Описание:**

1. ``sub`` - буквенный код сервера-отправителя.
2. ``aud`` - бувенный код сервера-получателя.
3. ``iat`` - время создания токена (в unix-значении).
4. ``exp`` - время истечения срока действия токена (в unix-значении).
5. ``role`` - тип отправителя.
6. ``type`` - тип запроса.

### Типы отправителя и запроса

> Информация актуальна на версию ``2.0.1-dev``.

Значение поля ``role`` строго типизировано. Допустимы следующие значения:

1. ``user`` - обращение пользователя через бота для работы конфиг-файла.
2. ``bot`` - обращение администратора через бота или веб-приложение для получения статистики, данных авторизаций и работы с конфиг-файлами пользователей.

Значение поля ``type`` также строго типизировано. Доступны следующие значения:

1. ``create`` - создать конфиг-файл. При ``role = user`` доступен только с HTTP-методом ``POST``.
2. ``recreate`` - пересоздать конфиг-файл. При ``role = user`` доступен только с HTTP-методом ``PUT`` или ``PATCH``.
3. ``drop`` - уничтожить конфиг-файл. При ``role = user`` доступен только с HTTP-методом ``DELETE``.
4. ``get`` - получить информацию о конфиг-файле. При ``role = user`` доступен только с HTTP-методом ``GET``.
5. ``check`` - получить информацию о состоянии сервера. Доступен **только** при ``role = bot``. Выполняется **только** с HTTP-методом ``POST``.

## Методы отслеживания состояния сервера

Для отслеживания состояния сервера и управления конфиг-файлами со стороны административного контроля над конфигурационными файлами, с версии ``2.0`` введены новые методы.

**Важно:** Актуально на версию ``2.0.2-dev``.

### Доступные запросы

#### Проверка сервера
Запрос ``/status`` - вызывается через HTTP-метод ``GET``. Выводит информацию о работе сервера и всех его вспомогательных программ.

```http
GET /v2.0/bot/status
```
**Пример ответа:** 

```json
    ...
    "data": {
        "date": "2026-01-28T16:37:16.032Z",
        "server_number": 1,
        "server_code": "ksd_nl_01",
        "isServerWorking": true,
        "isRedisRunning": true,
        "isOVPNDirExists": true,
        "isConfigsDirExists": true,
        "isOVPNActive": true
    },
    ...
```

**Описание:**

1. ``date`` - дата составления ответа и время (вплоть до миллисекунд) актуальности данных.
2. ``server_number`` - цифровой номер сервера.
3. ``server_code`` - код VPN-сервера (поле может быть пустым).
4. ``isServerRunning``: _boolean_ - работает ли сам сервер.
5. ``isRedisRunning``: _boolean_ - работает ли сервер Redis.
6. ``isOVPNDirExists``: _boolean_ - доступна ли папка-хранилище конфиг-файлов.
6. ``isConfigsDirExists``: _boolean_ - доступна ли папка-хранилище файлов пользователей.
7. ``isOVPNActive``: _boolean_ - активен ли сам процесс OpenVPN-сервера.

#### Проверка активности конфиг-файлов на данный момент

Запрос ``/active`` - вызывает через HTTP-метод ``GET``. Выводит информацию о подключённых конфиг-файлах на данный момент.

```http
GET /v2.0/bot/active
```

**Пример ответа:**

```json
{
    "date": "2026-01-28T17:53:55.673Z",
    "server_number": 1,
    "server_code": "ksd_nl_01",
    "active_users": 3
}
```

**Описание ответа:**

1. Описание пунктов ``date``, ``server_number`` и ``server_code`` является точно таким же, как и к предыдущему запросу

2. ``active_users``: _number_ - количество подключённых к серверу пользователей на данный момент.


## Пример ответов сервера

Все ответы сервера составляются по одному шаблону:

```json
{
    "code": 200,
    "message": "User configuration retrieved successfully",
    "data": {
        ...
    }
}
```

**Описание:**

1. ``code`` - числовой код ответа, согласно классификации ``HTTP``.
2. ``message`` - текстовое описание ответа.
3. ``data`` - данные, полученные по запросу. Может быть пустым полем.  

### Положительный ответ

Пример выполнен с запросом ``GET``:

```json
{
    "code": 200,
    "data": {
        "uuid": "430a8e06-d9b6-11f0-a8db-38f3ab6d0b2a",
        "version": 1,
        "user_type": "user",
        "created_at": "2024-06-26T12:00:00Z",
        "expired_time": "2024-07-26T12:00:00Z",
        "status": "active"
    },
    "message": "User configuration retrieved successfully"
}
```
### Негативный ответ

Пример выполнен с запросом ``GET``, но неверным ``UUID``:

```json
{
    "code": 400,
    "data": null,
    "message": "Invalid UUID format"
}
```
